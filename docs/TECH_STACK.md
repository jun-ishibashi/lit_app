# 技術スタックとプロダクトとの適合性

LIT APP の技術選定がプロダクトの要件に合っているかを整理したメモです。

---

## 1. 現在の技術スタック


| 層         | 技術                                              | 役割                                          |
| --------- | ----------------------------------------------- | ------------------------------------------- |
| **言語・FW** | Ruby 3.4, Rails 7.2                             | アプリ基盤。CRUD・認証・検索・ルーティング。                    |
| **フロント**  | HTML, CSS (SCSS), Bootstrap 5, Importmap, Turbo | マークアップ・スタイル・軽量 JS。SPA ではなくサーバー描画＋Turbo で遷移。 |
| **DB**    | 開発・テスト: SQLite 3 / 本番: PostgreSQL               | ローカルは簡易、本番は Render と相性のよい PostgreSQL。       |
| **認証**    | Devise                                          | User / Provider の二軸認証。                      |
| **検索**    | Ransack                                         | サービス一覧の絞り込み・ソート。                            |
| **その他**   | ActiveHash（マスタの一部）, Puma                        | マスタのコード管理、アプリサーバー。                          |
| **本番**    | Render                                          | ホスティング・PostgreSQL。                          |


---

## 2. プロダクトの要件（要約）

- **依頼側**: 出発地・到着地・作業範囲などで検索し、料金・リードタイムを比較。見積もり依頼。
- **業者側**: サービス登録・編集・見積もり依頼の確認。
- **性質**: B2B 寄り（中小企業の輸出支援）、検索・一覧・フォームが中心。リアルタイムチャットや SPA は不要。
- **規模**: 現状は小〜中規模を想定。大量トラフィック・複雑な非同期処理は想定外。

---

## 3. 適合している点


| 観点                              | 評価                                                                                |
| ------------------------------- | --------------------------------------------------------------------------------- |
| **Rails**                       | CRUD・認証・検索・フォーム・マイグレーションを一通り扱えており、開発速度と保守性のバランスがよい。LIT の「検索して比較・依頼する」という流れに合っている。 |
| **Devise**                      | User / Provider の二軸認証をそのまま活かせている。カスタムビューで「業者」表記に揃えるなどもしやすい。                       |
| **Ransack**                     | 出発地・到着地・サービスタイプ・作業範囲・業者・ソートを一つの `q` で扱えており、検索要件と一致している。                           |
| **Bootstrap 5**                 | フォーム・テーブル・カード・レスポンシブを少ないコードで揃えられる。業務寄り UI で十分。                                    |
| **Importmap + Turbo**           | npm や Node を本格導入せずに、Turbo で遷移を快適にできる。SPA にする必要がないため、スタックがシンプルで済んでいる。              |
| **SQLite（開発） / PostgreSQL（本番）** | ローカルはセットアップが軽く、本番は Render と相性がよい。                                                 |
| **Render**                      | Rails のデプロイがしやすく、PostgreSQL も連携しやすい。小〜中規模のサービスに適している。                             |


---

## 4. 注意・検討するとよい点


| 観点            | 内容                                                                                                                             |
| ------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| **検索のスケール**   | サービス数が大きく増えた場合、Ransack の `limit(100)` やインデックス（departure_id, destination_id 等）で十分か、全文検索や Elasticsearch が必要になるかは、データ量に応じて見直すとよい。 |
| **フロントの複雑化**  | 将来的に「ドラッグで並び替え」「リアルタイム通知」などが欲しくなった場合は、Stimulus の強化や React/Vue の部分導入を検討する余地がある。現状の要件なら不要。                                       |
| **Importmap** | JS 依存が増えていくと、Bundler や esbuild に移行した方が管理しやすい場合がある。現状は軽量なので問題なし。                                                                |
| **テスト**       | Minitest と Rails 7 の組み合わせで、環境によってはテストランナーの互換性に注意。CI で安定して回すことが重要。                                                              |


---

## 5. まとめ

- **現状のプロダクト（検索・比較・見積もり依頼・業者の登録・一覧）には、Rails + Devise + Ransack + Bootstrap + Turbo のスタックはよく合っている。**
- サーバーサイド中心で、SPA やリアルタイム性を求めていないため、スタックを重くせずに開発・運用しやすい構成になっている。
- スケールや機能拡張（検索の高度化、フロントのリッチ化）が必要になった段階で、該当する層だけ選定を見直せばよい。

## 6. Favicon

- **設定**: `public/favicon.svg` を配置し、レイアウトで `<link rel="icon" type="image/svg+xml" href="/favicon.svg" />` を出力している。
- デザインはヘッダーロゴ（角丸四角・L 型・丸）をベースに、アプリのプライマリ色（`--lit-primary: #2563eb`）で統一。
- 必要に応じて [realfavicongenerator.net](https://realfavicongenerator.net/) 等で `favicon.ico` や Apple Touch Icon を生成し、同じく `public/` に置いてリンクを追加できる。
