# CargoLink — ビジネスロードマップ

海外輸出の輸送手段を業者ごとに料金・リードタイムで比較する「輸送の乗換案内」を目指す CargoLink のビジネスロードマップです。フェーズごとに「いま」「近い将来」「中期」「長期」で整理しています。

---

## ビジネス概要

| 項目 | 内容 |
|------|------|
| **課題** | 日本の生産者（特に中小企業）が海外にモノを輸出する際、どの輸送手段・業者を選べばよいか判断しづらい。 |
| **提供価値** | 複数業者の相見積もりを一括で比較できる Web サービス。 |
| **主な利用者** | **依頼側（ユーザー）**: 料金・リードタイムの検索・比較。**提供側（業者）**: 自社サービスの登録・管理。 |

## Phase 0：現在（基盤の安定化・UX の整理）

**目的**: 運用で困らない土台と、依頼側・業者側双方が迷わない体験に整える。

### 施策

| 項目 | 内容 | 参照 |
|------|------|------|
| 認証の抜け解消 | ユーザー・業者のマイページを「要ログイン」にし、未ログイン時はログイン画面へ。 | DESIGN_AND_IMPROVEMENTS |
| 業者 show の設計確定 | 「マイページ専用」なら専用パス（例: `/mypage/provider`）、「他社も見る」なら `params[:id]` で表示し編集は自社のみ。 | DESIGN_AND_IMPROVEMENTS |
| ヘッダー・フッターのレイアウト化 | 全ページ共通のヘッダー・フッターをレイアウトに寄せ、各ビューからの重複 render を削除。 | DESIGN_AND_IMPROVEMENTS |
| 不要な @services の整理 | サービス詳細で使っていない `@services` をコントローラから削除。 | DESIGN_AND_IMPROVEMENTS |

### UX・用語

| 項目 | 内容 | 参照 |
|------|------|------|
| 表記の統一 | 画面・ドキュメントで「プロバイダー」→「業者」「業者様」に寄せる。コードは `Provider` のまま。 | PROVIDER_UX_AND_TERMINOLOGY |
| ナビの整理 | 業者向けを「業者様はこちら」1リンクにまとめ、その先でログイン/登録を選択。 | PROVIDER_UX_AND_TERMINOLOGY |
| ページタイトル | 主要ページで `content_for :title` を設定し、タブ・ブックマークで区別しやすくする。 | DESIGN_AND_IMPROVEMENTS |

### 品質・運用

| 項目 | 内容 |
|------|------|
| テストの充実 | 認可（他社サービスの編集拒否）、未ログイン時のリダイレクト、検索パラメータまわりをテストで担保。 |
| 文言の統一 | 「該当する商品」→「該当するサービス」など、サービス名に合わせて統一。 |

## Phase 1：短期（〜3ヶ月）— 信頼性・導線・用語の統一

**目的**: 「業者として使いやすい」「依頼側が迷わない」状態にし、運用リスクを下げる。

###  product

| 項目 | 内容 |
|------|------|
| 業者入口の明確化 | 「業者様はこちら」先のページを整備。ログイン後は業者マイページをホームに見せる導線に。 |
| 業者マイページの強化 | サービス一覧・登録・編集への導線をマイページに集約。「管理画面」としての体感に近づける。 |
| 404/500 ページ | 本番用のエラーページを用意し、レイアウト・文言をアプリのトーンに合わせる。 |

### データ・マスタ

| 項目 | 内容 |
|------|------|
| マスタ typo 修正 | 出発地・到着地の typo（例: Barselona→Barcelona, Penan→Penang）をコード上で修正。 |
| マスタの扱い方針 | 出発地・到着地の増減が少ないうちは ActiveHash のまま。変更頻度が増えたら DB 化を検討。 |

### セキュリティ・運用

| 項目 | 内容 |
|------|------|
| 本番シークレット | `RAILS_MASTER_KEY` 等の環境変数管理と漏洩防止の確認。 |
| バックアップ | Render（本番）の PostgreSQL バックアップ方針の確認・設定。 |
| CI | GitHub Actions 等でテスト・Lint の自動実行を導入。 |

## Phase 2：中期（3〜6ヶ月）— 料金の実態に近づける・つながりを作る

**目的**: 「見た目の運賃」から「実務に近い比較」へ近づけ、依頼側と業者をつなぐ第一歩を用意する。

###  product

| 項目 | 内容 |
|------|------|
| 料金項目の見直し | 運賃のみ→サーチャージ・通関費・保険等の項目追加や「概算」表示の検討。実際の輸送コストとの乖離を減らす。 |
| 相談・問い合わせ | サービス詳細の「相談する」を実装。メール送信・フォーム連携・簡易チャットのいずれかで依頼→業者への連絡を可能に。 |
| 数量・重量と料金 | ルート単位の固定料金に加え、重量・CBM 等を入力して料金を計算する機能の検討。 |
| 日付計算の高度化 | 希望出荷日・到着日について、休業日・船便スケジュールを考慮した表示の検討（可能な範囲で）。 |

### データ・モデル

| 項目 | 内容 |
|------|------|
| ユーザーとサービスの関係 | 見積依頼・問い合わせ履歴などを残す場合、User と Service の中間モデル（見積・問い合わせ等）の検討。 |
| マスタの DB 化（検討） | 出発地・到着地の増減・名前変更が頻繁になる場合、管理画面または DB マスタ化を検討。 |

### 業者まわり

| 項目 | 内容 |
|------|------|
| 業者審査・公開制御 | 登録すれば即公開ではなく、審査フローや「下書き／公開」の検討。 |
| スパム・不正対策 | 大量登録・不正料金へのレート制限・モデレーション等の検討。 |

---

## Phase 3：長期（6ヶ月〜）— 取引の記録・評価・マスタの柔軟化

**目的**: 比較だけでなく「選んだ結果」を残し、信頼指標と運用のしやすさを高める。

###  product

| 項目 | 内容 |
|------|------|
| レビュー・評価 | 利用後のレビューや、サービス・業者への評価機能の検討。 |
| 成約・履歴 | 見積→成約までの流れをアプリ内に持つ場合、User–Service の中間テーブル（注文・成約）の拡張。 |
| 業者向けダッシュボード | 業者数・問い合わせ数が増えたタイミングで、`/for-providers` 先の専用エリアを検討。問い合わせ一覧・簡易分析など。 |

### データ・インフラ

| 項目 | 内容 |
|------|------|
| マスタ管理 | 出発地・到着地・オプション等の管理画面または API による運用。 |
| ログ・監視 | 本番のエラー検知・パフォーマンス監視の導入。 |
