# LIT APP - ビジネスロジックと今後の検討事項

## 1. 全体のビジネスロジック

### 1.1 役割

| 役割 | モデル | 説明 |
|------|--------|------|
| **ユーザー** | User | 輸送を依頼する側（法人・個人）。出発地・到着地でサービスを検索し、料金・リードタイムを比較する。 |
| **プロバイダー** | Provider | 運送業者。自社のサービス（ルート・料金・リードタイム・オプション）を登録・編集・削除する。 |
| **サービス** | Service | 1つの「ルート＋輸送手段＋料金＋リードタイム」の組み合わせ。プロバイダーに紐づく。 |

### 1.2 主な流れ

1. **ユーザー（未ログイン可）**
   - トップ（`/`）でサービス一覧（直近10件）を表示。
   - 出発地・到着地・サービスタイプ・プロバイダー・並び順で検索（`/services/search`）。
   - 希望出荷日・希望到着日を指定すると、検索結果に「最短到着日」「必要出荷日」を表示。
   - サービス詳細（`/services/:id`）で内容を確認。「相談する」は未実装（リンクのみ）。

2. **プロバイダー（要ログイン）**
   - ログイン後、トップから「新しいサービスを登録」でサービス作成。
   - 自分のサービスのみ編集・削除可能（他社は権限なしでリダイレクト）。

3. **マスタ（ActiveHash）**
   - 出発地（Departure）・到着地（Destination）・サービスタイプ（航空/船舶LCL/FCL）・オプション（通常品/危険品/要冷蔵等）・ユーザータイプ（法人/個人）・製品（Product）はコードで定義され、DB には持たない。

### 1.3 技術的なポイント

- **検索**: Ransack で `departure_id_eq`, `destination_id_eq`, `service_type_id_eq`, `provider_id_eq`, `sorts`（価格順・リードタイム順）を指定。
- **認証**: Devise で User と Provider を別々に管理（同時ログインは想定していない運用）。
- **認可**: サービスの作成・編集・削除はプロバイダーのみ。編集・削除は「そのサービスの provider_id == current_provider.id」のときのみ許可。

## 2. 今後の検討事項

### 2.1 機能・仕様

| 項目 | 内容 |
|------|------|
| **料金の妥当性** | 現状は運賃のみ。サーチャージ・通関費・保険等を含めると実際コストと乖離するため、項目の追加や「概算」表示の検討。 |
| **相談・問い合わせ** | 詳細画面の「相談する」は未実装。メール送信・フォーム連携・チャット等の検討。 |
| **レビュー・評価** | README で言及されている「利用後のレビュー」は未実装。Service または Provider への評価機能の検討。 |
| **数量・重量による料金** | 現状はルート単位の固定料金。重量・CBM 等を入力して料金を計算する機能の検討。 |
| **ユーザーとサービスの関係** | 現状、User はサービスを「検索して見る」だけ。見積依頼・成約履歴などを残すなら、User と Service の中間テーブル（見積・注文など）の検討。 |

### 2.2 データ・マスタ

| 項目 | 内容 |
|------|------|
| **出発地・到着地** | 現在は ActiveHash で固定。増減や名前修正のたびにコード変更が必要。管理画面や DB マスタ化の検討。 |
| **Destination の typo** | `Barselona` → `Barcelona`、`Penan` → `Penang` 等の修正検討。 |
| **Option の「--」** | Service の option_id は必須。ActiveHash の Option に id:1 が「通常品」のため、未選択用「--」は別扱い（現状は問題なし）。 |

### 2.3 セキュリティ・運用

| 項目 | 内容 |
|------|------|
| **本番の master.key** | Render 等では `RAILS_MASTER_KEY` を環境変数で設定。漏洩に注意。 |
| **プロバイダー審査** | 現状は登録すれば誰でもサービス登録可能。審査フローや「公開/下書き」の検討。 |
| **スパム・不正** | 大量登録・不正料金の対策（レート制限・モデレーション等）の検討。 |

### 2.4 UX・UI

| 項目 | 内容 |
|------|------|
| **検索結果 0 件** | メッセージは「該当する商品はありません」→「該当するサービスはありません」等の文言統一検討。 |
| **日付指定** | 希望出荷日・到着日は「日数」で計算しているだけ。休業日・船便スケジュールは未考慮。 |
| **モバイル** | Bootstrap でレスポンシブにはなっているが、実機での確認推奨。 |

### 2.5 インフラ・開発

| 項目 | 内容 |
|------|------|
| **CI** | テスト・Lint の自動実行（GitHub Actions 等）の検討。 |
| **本番 DB バックアップ** | Render の PostgreSQL バックアップ方針の確認・設定。 |
| **ログ・監視** | 本番エラー検知・パフォーマンス監視の検討。 |

## 3. シードデータの使い方

開発環境でモック・テストデータを投入する場合:

```bash
bundle exec rails db:seed
```

全消ししてマイグレーション＋シードからやり直す場合:

```bash
bundle exec rails db:reset
```
